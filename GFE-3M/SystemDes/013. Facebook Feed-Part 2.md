# News Feed Frontend System Design (Part 2)

# Rich Text Composer (WYSIWYG)

## What is WYSIWYG?

WYSIWYG stands for:

> What You See Is What You Get

The editor visually resembles the final rendered post while the user is typing.

Examples:

* Mentions appear highlighted
* Hashtags appear styled
* Links appear formatted
* Rich formatting is visible immediately

---

# Rich Text Mentions and Hashtags

A feed composer ideally supports:

* Mentions (@user)
* Hashtags (#topic)
* Links
* Rich text formatting

while maintaining a visual editing experience.

---

## Why `<textarea>` Is Not Enough

Traditional inputs only support plain text:

```html
<input />
<textarea></textarea>
```

Limitations:

* No rich formatting
* No mention highlighting
* No embedded entities
* No advanced selection model

---

## contenteditable

HTML provides:

```html
<div contenteditable="true"></div>
```

which allows rich-text editing.

---

### Why Not Use contenteditable Directly?

Using raw `contenteditable` in production introduces many issues:

* Selection bugs
* Cursor management complexity
* Undo/redo inconsistencies
* Browser-specific behavior
* Mobile keyboard edge cases
* Paste handling problems
* Accessibility challenges

---

## Recommended Approach

Use battle-tested editor frameworks.

Popular choices:

### Lexical

Created by Meta.

Advantages:

* Modern architecture
* Excellent performance
* Plugin ecosystem

---

### TipTap

Built on ProseMirror.

Advantages:

* Rich feature set
* Excellent customization

---

### Slate

Advantages:

* Flexible data model
* React-first architecture

---

### Draft.js

Originally developed by Meta.

Status:

* Deprecated
* Not recommended for new projects

# Internal Editor Model vs Storage Model

A rich-text editor often uses an internal tree structure.

Example responsibilities:

* Selection tracking
* Undo/redo history
* Cursor movement
* Rich formatting

The API and database typically benefit from a much simpler representation.

---

## Recommended Architecture

Maintain:

### Editor Model

Rich tree structure for editing.

### Storage Model

Compact serialized structure.

Convert between them at boundaries.

Benefits:

* Smaller payloads
* Easier rendering
* Easier backend storage
* Platform independence

The same serialized representation can be rendered on:

* Web
* iOS
* Android

without sharing editor implementation details.

---

# Lazy-Loaded Composer Features

Many composer tools are rarely opened.

Loading them in the initial bundle wastes bandwidth.

---

## Good Lazy-Loading Candidates

### Image Uploader

Load only when user selects media.

---

### GIF Picker

Often contains:

* Large UI bundles
* External integrations

Load on demand.

---

### Emoji Picker

Can be surprisingly large.

Load only when opened.

---

### Sticker Picker

Good interaction-triggered loading candidate.

---

### Background Images

Only required for users who customize post appearance.

---

## Why Lazy Load?

Many users never use these features.

Benefits:

* Smaller initial bundle
* Faster startup
* Better LCP
* Lower memory usage

# Networking, Resilience, Offline, and Retries

Real users:

* Lose connectivity
* Change networks
* Close tabs
* Experience flaky internet

The application must remain reliable under all conditions.

---

# Error States

Network failures should always be visible.

Examples:

* Failed feed loading
* Failed reaction
* Failed post submission
* Offline mode

Never silently fail.

---

# Reading Offline

## Service Worker

A Service Worker forms the foundation of offline support.

---

### Cached Assets

Cache:

* Application shell
* Critical JS
* CSS
* Static assets

Benefits:

* Instant repeat visits
* Offline startup

---

### Cached Feed Content

Cache:

* Recent feed pages
* Recently viewed media

A user opening the app without connectivity should still see:

* Previously viewed feed
* Cached posts
* Cached images

with clear indications that content may be stale.

---

## IndexedDB Cache Persistence

Libraries such as:

* TanStack Query

can persist cached data into IndexedDB.

Benefits:

* Survives reloads
* Survives tab closure
* Enables instant cache rendering

Typical flow:

1. Render cached data
2. Revalidate in background
3. Update UI if newer data exists

---

# Offline Writes with an Outbox

Reading offline is easy.

Writing offline is harder.

---

## Problem

User:

* Creates post
* Reacts
* Shares

Connection disappears.

We must not lose the action.

---

## Outbox Pattern

When user performs a write:

### Step 1

Store mutation in IndexedDB.

```text
Outbox
```

---

### Step 2

Apply optimistic update.

---

### Step 3

Attempt network request.

---

### Success

Remove outbox item.

---

### Failure

Keep item.

Retry later.

---

# Outbox Keying

Each mutation should be stored using:

```text
Idempotency Key
```

Benefits:

* Safe retries
* Duplicate prevention
* Crash recovery

---

# Background Sync

The Service Worker can retry writes even after:

* Tab closes
* Browser reloads

using:

```text
Background Sync API
```

---

## Caveat

Browser support is inconsistent.

Notably:

* Safari lacks support

Treat it as:

> Best-effort optimization

not correctness.

---

## Correctness Path

Maintain:

* In-tab retry mechanism
* Persistent outbox

regardless of Background Sync support.

---

# Retry Strategy

## Exponential Backoff

Standard approach:

```text
1s
2s
4s
8s
16s
```

---

## Add Jitter

Randomize retry times.

Example:

```text
4.3s
7.8s
15.2s
```

instead of exact powers of two.

---

## Why Jitter?

Prevents:

```text
Thundering Herd
```

Millions of clients reconnecting simultaneously after an outage.

---

# Retry Limits

Never retry forever.

Set:

* Maximum attempts
* Retry window

Eventually surface error to user.

---

# Non-Retryable Errors

Do not retry:

```text
400
401
403
404
422
```

because they are unlikely to succeed later.

---

# Combining Reliability Techniques

Safe write path consists of:

* Idempotency keys
* Request deduplication
* Outbox persistence
* Retry logic
* Background sync

This makes writes safe under:

* Client retries
* Service Worker retries
* Network retries

The server always sees:

```text
Same idempotency key
```

and returns:

```text
Same result
```

# Live Updates

Common approaches:

1. Short Polling
2. Long Polling
3. SSE
4. WebSockets

---

## Short Polling

Client repeatedly requests updates.

Example:

```text
Every 10 seconds
```

Advantages:

* Easy implementation

Disadvantages:

* High network traffic
* High server load

---

## Long Polling

Server keeps request open.

Returns only when updates exist.

Advantages:

* Simpler than WebSockets

Disadvantages:

* More latency
* More server resources

---

## Server-Sent Events (SSE)

Server pushes updates through a persistent HTTP connection.

Advantages:

* Simple
* Efficient
* Built into browsers

Best for:

Server → Client communication

---

## WebSockets

Full-duplex connection.

Advantages:

* Low latency
* Bidirectional
* Highly interactive

Best for:

* Reactions
* Chat
* Live engagement

---

# High-Traffic Posts

Examples:

* Celebrities
* Politicians
* Breaking news

Do not push every individual event.

Instead:

### Count-Only Updates

Push:

```text
10,523 reactions
```

instead of:

```text
User A reacted
User B reacted
User C reacted
...
```

Benefits:

* Lower CPU
* Lower bandwidth
* Better readability

# Performance and Metrics

Feed performance must be measured from the user's perspective.

---

## Core Web Vitals

### LCP

Largest Contentful Paint

Measures:

> How quickly meaningful content appears.

---

### INP

Interaction to Next Paint

Measures:

> Responsiveness of user interactions.

Examples:

* Reactions
* Menus
* Scrolling
* Composer interactions

---

### CLS

Cumulative Layout Shift

Measures:

> Visual stability

Examples:

* Image loading
* Feed hydration
* Lazy-loaded content

---

# Most Important Metric

After initial load:

### INP

is usually more important than LCP.

Reason:

Feed sessions may last:

* Tens of minutes
* Thousands of interactions

A responsive experience matters more than startup alone.

# Facebook's Loading Philosophy

Facebook divides loading into tiers.

---

## Tier 1

Display pixels immediately.

---

## Tier 2

Complete visible feed.

---

## Tier 3

Everything else.

Examples:

* Hover cards
* Menus
* Advanced tools

---

# Code Splitting Strategy

## Route-Level Splitting

Load routes only when needed.

Examples:

* Profile
* Notifications
* Settings

---

## Data-Driven Splitting

Load renderer matching feed content.

Example:

Only load image renderer if image posts exist.

---

## Interaction-Based Splitting

Load features only after user intent.

Examples:

* Emoji picker
* Reaction picker
* Hover cards

---

## Idle-Time Prefetching

Prefetch likely interactions during idle time.

Examples:

* Hover intent
* Idle browser periods

---

# LCP vs INP Tradeoff

Code splitting improves:

```text
LCP
```

But may hurt:

```text
INP
```

if a click waits for code download.

Solution:

Prefetch common interactions after initial render.

---

# Main Thread Protection

Keep expensive work off main thread.

Examples:

* Parsing large payloads
* Rich text rendering
* Mention extraction
* Media processing

---

## Web Workers

Move heavy work to background threads.

---

## scheduler.postTask()

Split long work across frames.

Improves responsiveness.

# Internationalization (i18n)

---

## RTL Layouts

Languages:

* Arabic
* Hebrew
* Persian

Use:

```css
margin-inline-start
padding-inline-end
```

instead of:

```css
margin-left
padding-right
```

Benefits:

Automatic RTL support.

---

## Bidirectional Text

Posts often mix languages.

Use:

```html
dir="auto"
```

so browser determines direction automatically.

---

## Pluralization

Avoid:

```js
count === 1 ? 'like' : 'likes'
```

Use:

```js
Intl.PluralRules
```

and ICU MessageFormat.

Many languages have more than two plural forms.

---

## Number Formatting

Use:

```js
Intl.NumberFormat
```

Examples:

* 103K
* 103 тыс.
* 10.3萬

---

## List Formatting

Use:

```js
Intl.ListFormat
```

for localized conjunctions.

---

## RTL Recommendation

Use logical CSS properties from day one.

Retrofitting RTL later is expensive and error-prone.

# Final Summary

A Facebook-style news feed is a long-lived, personalized, interaction-heavy application.

The design prioritizes:

* Fast initial rendering
* Smooth scrolling
* Reliable writes
* Offline support
* Accessibility
* Internationalization
* Real-user performance

Key principles:

* CSR + SPA by default for authenticated feeds
* Normalized client-side store
* Cursor-based pagination
* Virtualized feed rendering
* Optimistic updates with idempotent writes
* Offline outbox and retry support
* Data-driven code loading
* Accessibility-first interactions
* Real-user monitoring and gradual rollout

These patterns allow the feed to remain responsive, resilient, and scalable even during long sessions, poor network conditions, high traffic events, and global usage across many devices and languages.
