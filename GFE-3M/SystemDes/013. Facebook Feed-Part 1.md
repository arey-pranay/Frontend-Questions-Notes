# News Feed Frontend System Design

## Problem Statement

Design a web news feed application that allows users to:

* Browse a feed of posts
* React to posts
* Create new posts

### Requirements

* Render the initial feed quickly
* Load additional posts as users scroll
* Support reactions and post creation
* Handle long-lived sessions
* Handle stale data
* Address common feed performance concerns

### Out of Scope

* Feed ranking algorithms
* Ads delivery systems
* Deep backend fan-out / fan-in architectures

---

# Rendering Strategy

Before designing the frontend architecture, two foundational decisions must be made:

1. Where rendering happens
2. How navigation works after initial page load

---

## Rendering Options

### Server-Side Rendering (SSR)

The server generates complete HTML before sending it to the browser.

#### Advantages

* Faster first paint
* Better SEO
* Crawlers receive meaningful content immediately

#### Tradeoffs

* Hydration required before interactivity
* More server work

---

### Client-Side Rendering (CSR)

The server returns:

* Minimal HTML shell
* JavaScript bundles

The browser then:

* Fetches data
* Renders UI

#### Advantages

* Excellent interactivity
* No full-page reloads
* Long-lived client state

#### Tradeoffs

* Slower cold start
* Weaker SEO

---

### Hybrid Rendering

Combines SSR and CSR.

#### Flow

1. Initial page rendered on server
2. Hydration occurs
3. Subsequent updates handled client-side

---

## Recommended Choice: CSR for Signed-In Feed

For a personalized feed:

SEO is less important because content is user-specific.

The primary goal becomes:

* Responsiveness
* Session continuity
* Rich client-side interactions

Therefore:

### Use CSR for the signed-in home feed

This recommendation is based on state-management benefits, not cold-start speed.

To maintain competitive LCP:

* Minimize app-shell size
* Keep critical JS small
* Limit initial feed payload

SSR/Hybrid can still be useful for:

* Public post pages
* Marketing pages
* Logged-out experiences

---

# Navigation Strategy

## SPA vs MPA

### Single Page Application (SPA)

Recommended.

#### Why?

Most users open post details from the feed.

The feed already contains:

* Post text
* Media metadata
* Author information

Therefore:

Navigating to post details can reuse existing state and only fetch:

* Comments
* Replies
* Missing details

#### Benefits

* Near-instant navigation
* Shared cache
* Preserved scroll position
* Preserved drafts
* Preserved optimistic updates

---

### Multi Page Application (MPA)

Each navigation:

* Destroys current page
* Reloads data
* Recreates state

This loses:

* Cached entities
* Drafts
* Scroll state
* Optimistic updates

# Frontend Architecture

The frontend is separated into four layers.

```
View
 ↓
Store
 ↓
Data Access Layer
 ↓
Server
```

---

## 1. View Layer

Responsible for:

* Feed page
* Post detail page
* Feed posts
* Post composer

### Responsibilities

* Render state
* Handle user interactions
* Trigger actions

Examples:

* Reacting to a post
* Creating a post

---

## 2. Store Layer

Client-side source of truth.

Stores:

* Feed data
* Posts
* Users
* Media
* Drafts
* Optimistic updates
* Freshness metadata

### Responsibilities

* Maintain consistency
* Enable responsive UI
* Decouple UI from network latency

---

## 3. Data Access Layer

Abstracts API communication.

Responsibilities:

* Network requests
* Response parsing
* Pagination
* Retry logic
* Caching
* Data transformations

Examples:

* RTK Query
* TanStack Query
* Relay
* Apollo Client

---

## 4. Server Layer

Provides canonical data.

Endpoints:

* Fetch feed
* Fetch post
* Create post
* Upload media
* React to post
* Share post

---

## Benefits of Layered Architecture

### View

Focuses on presentation.

### Store

Manages application state.

### Data Layer

Handles synchronization.

### Server

Owns canonical data.

# Core Entities

## Post

```ts
type PostBody = {
  text: string;
  entities: Array<{
    type: 'mention' | 'hashtag' | 'link';
    start: number;
    end: number;
    userId?: string;
    url?: string;
  }>;
};

type ReactionType =
  | 'like'
  | 'love'
  | 'haha'
  | 'wow'
  | 'sad'
  | 'angry';

type EngagementSummary = {
  reactions: Record<ReactionType, number>;
  totalReactions: number;
  commentCount: number;
  shareCount: number;
};

type Post = {
  id: string;
  authorId: string;
  body: PostBody;
  mediaIds: string[];
  engagementSummary: EngagementSummary;
  viewerReaction: ReactionType | null;
  viewerHasShared: boolean;
  createdAt: number;
};
```

---

## User

```ts
type RelationshipToViewer = {
  isFriend?: boolean;
  isFollowing?: boolean;
  isMuted?: boolean;
  isBlocked?: boolean;
};

type User = {
  id: string;
  name: string;
  handle: string;
  profilePhotoUrl: string;
  isVerified: boolean;
  relationshipToViewer: RelationshipToViewer;
};
```

### Important Design Principle

User data should be globally normalized.

Do NOT duplicate user objects inside posts.

Benefits:

* Single source of truth
* Consistent updates
* Reduced memory usage
* Reduced network requests

---

## Feed

```ts
type Feed = {
  id: string;
  postIds: string[];
  olderCursor: string | null;
  newerCursor: string | null;
  hasOlder: boolean;
  hasNewer: boolean;
  lastFetchedAt: number | null;
};
```

### Key Insight

A feed is:

> An ordered list of post IDs plus metadata

Not:

> A deeply nested list of full post objects

This enables:

* Efficient updates
* Top insertion
* Bottom append
* Partial refreshes

# Pagination

## Offset Pagination

Example:

```http
/feed?offset=20&limit=10
```

### Advantages

* Simple
* Easy to implement

### Problems

Offsets shift when new posts appear.

Can cause:

* Missing posts
* Duplicate posts

Best for:

* Search results
* Static lists

---

## Cursor Pagination

Example:

```http
/feed?cursor=POST_ID
```

### Advantages

* Stable
* Efficient
* Handles inserts
* Handles deletions
* Handles reordering

### Ideal For

Dynamic feeds

---

## Why Cursor Pagination Wins

Feeds constantly change.

New posts:

* Appear
* Disappear
* Re-rank

Cursor pagination avoids inconsistency while scaling efficiently.

---

## Bidirectional Pagination

Supports:

### Older Content

Append to bottom.

### Newer Content

Insert at top.

Without:

* Refetching everything
* Missing content

# Write Reliability

## Idempotency Keys

Required for:

* Post creation
* Reactions
* Shares

### Why?

Requests may be retried by:

* Client
* Service Worker
* Proxy
* Network layer

### Rule

Generate key when user submits.

Not when request is sent.

Example:

```http
Idempotency-Key: uuid
```

Server should return original result for duplicate keys.

For posts with media:

### Step 1

### Step 2

Create post

# Feed List Optimization

## Virtualization

Long feeds create:

* Huge DOM trees
* Slow layout
* High memory usage

### Solution

Render:

* Visible items
* Small overscan region

Everything else becomes:

```html
<div style="height:300px"></div>
```

### Benefits

#### Browser

Less painting

#### Layout

Less computation

#### React

Smaller reconciliation cost

### Requirements

* Stable keys
* Accurate height measurement

---

# Infinite Scrolling

* Early enough

# Multi-Tab Consistency

Problem:

User reacts in Tab A.

Tab B becomes stale.

### Solution

BroadcastChannel

Mutating tab broadcasts updates.

Other tabs refresh entities.

### Advanced

IndexedDB + Web Locks

Leader tab:

* Polling
* WebSocket ownership

Follower tabs:

* Shared cache reads

# Timestamps

## Option 1

Server returns raw timestamp.

Client formats.

### Pros

Flexible

### Cons

More JS

---

## Option 2

Server returns translated string.

### Pros

Smaller client

### Cons

Less flexibility

---

## Recommended

Use:

```js
Intl.DateTimeFormat()
Intl.RelativeTimeFormat()
```

---

## Long Sessions

Refresh recent timestamps periodically.

Example:

```
2 minutes ago
```

should not remain unchanged for hours.

Older timestamps can remain static.

# Icon Rendering

| Approach    | Pros        | Cons                            |
| ----------- | ----------- | ------------------------------- |
| Image       | Simple      | Many requests                   |
| Spritesheet | One request | Setup complexity                |
| Icon Font   | Scalable    | Accessibility issues, FOIT/FOUT |
| SVG         | Scalable    | Download flicker                |
| Inline SVG  | Scalable    | Cannot be cached separately     |

### Industry Trend

Facebook and Twitter primarily use inline SVG.
