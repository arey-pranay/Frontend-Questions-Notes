# 📄 Understanding `resolve`, `reject`, `.then`, `.catch` & Promise Internals

This is one of the MOST important Promise concepts.

You’ve touched the exact thing interviewers love asking:

> “What exactly ARE `resolve` and `reject`?”

and:

> “Why can we pass `reject` directly into `.catch()`?”

Let’s fully break it down.

---

# 1️⃣ What ARE `resolve` and `reject`?

When you create a promise:

```js id="r11"
new Promise((resolve, reject) => {

});
```

JavaScript AUTOMATICALLY gives you:

* a `resolve` function
* a `reject` function

---

# 🚨 They are NORMAL FUNCTIONS

They are NOT keywords.

NOT special syntax.

Just functions provided by the Promise constructor.

---

# Think of it like:

```js id="r12"
function myPromiseConstructor(resolve, reject) {

}
```

JS internally calls your executor function with these functions.

---

# 2️⃣ What do they do?

---

# `resolve(value)`

Marks promise as:

```text id="r13"
fulfilled
```

and stores:

* resolved value

---

# Example

```js id="r14"
const p = new Promise((resolve) => {
  resolve(5);
});
```

Now:

```text id="r15"
p → fulfilled with value 5
```

---

# `reject(error)`

Marks promise as:

```text id="r16"
rejected
```

and stores:

* rejection reason/error

---

# Example

```js id="r17"
const p = new Promise((_, reject) => {
  reject("failed");
});
```

---

# 3️⃣ Why Can We Pass `reject` Into `.catch()`?

This line:

```js id="r18"
.catch(reject)
```

looks magical.

But it’s actually VERY simple.

---

# `.catch()` expects a function

Equivalent to:

```js id="r19"
.catch((err) => {
  reject(err);
})
```

---

# Since `reject` ITSELF is already a function:

we can pass it directly.

---

# EXACT SAME THING

These are equivalent:

---

# Version 1

```js id="r20"
.catch(reject)
```

---

# Version 2

```js id="r21"
.catch((err) => reject(err))
```

---

# 🚨 SUPER IMPORTANT JS CONCEPT

Functions are:

```text id="r22"
first-class citizens
```

Meaning:

* stored in variables
* passed as arguments
* returned from functions

---

# Example

```js id="r23"
function hello() {}

const x = hello;
```

`x` now references function itself.

---

# Same idea here

```js id="r24"
.catch(reject)
```

passes function reference.

---

# 4️⃣ `.then()` and `.catch()` ALSO Take Functions

---

# Example

```js id="r25"
promise.then(onSuccess);
```

Equivalent to:

```js id="r26"
promise.then((value) => {
  onSuccess(value);
});
```

---

# Same for catch

```js id="r27"
promise.catch(onError);
```

---

# 5️⃣ Promise Constructor Flow

VERY IMPORTANT.

---

# Example

```js id="r28"
const p = new Promise((resolve, reject) => {

  setTimeout(() => {
    resolve(10);
  }, 1000);

});
```

---

# What Happens?

---

# Step 1

Promise created.

State:

```text id="r29"
pending
```

---

# Step 2

Executor function runs IMMEDIATELY.

Very important.

---

# Step 3

`setTimeout` scheduled.

---

# Step 4

After 1 second:

```js id="r30"
resolve(10)
```

called.

---

# Step 5

Promise becomes:

```text id="r31"
fulfilled
```

with value:

* 10

---

# 6️⃣ `.then()` Internally

When you do:

```js id="r32"
promise.then(fn)
```

Promise internally stores:

* callback

When promise resolves:

* callback pushed to microtask queue

---

# 🚨 Important

`.then()` does NOT execute immediately.

Even if promise already resolved.

---

# Example

```js id="r33"
Promise.resolve(5)
  .then(console.log);

console.log("end");
```

Output:

```text id="r34"
end
5
```

Because `.then` callbacks are microtasks.

---

# 7️⃣ Promise State Machine

Promises internally track:

---

# State

```text id="r35"
pending
fulfilled
rejected
```

---

# Value

```text id="r36"
resolved data OR error
```

---

# Callback Queues

Stored `.then()` handlers.

---

# 8️⃣ Why Promise Constructor Exists

It lets YOU convert callback APIs into promises.

---

# Example

Old callback API:

```js id="r37"
setTimeout(callback, 1000);
```

Promise wrapper:

```js id="r38"
function wait(ms) {
  return new Promise((resolve) => {
    setTimeout(resolve, ms);
  });
}
```

---

# 9️⃣ `resolve` Can Accept Promises Too

VERY IMPORTANT.

---

# Example

```js id="r39"
resolve(fetch("/api"))
```

Promise automatically adopts inner promise state.

Called:

```text id="r40"
Promise flattening
```

or:

```text id="r41"
Promise Resolution Procedure
```

---

# 🔟 Why `.catch(reject)` Works PERFECTLY in Your Polyfill

Your code:

```ts id="r42"
Promise.resolve(item)
  .then(...)
  .catch(reject);
```

means:

---

# If item rejects

↓

call outer promise's reject function

↓

reject entire `Promise.all`

Exactly native behavior.

---

# 1️⃣1️⃣ Promise Constructor Anti-Pattern

VERY IMPORTANT interview topic.

---

# BAD

```js id="r43"
return new Promise(async (resolve) => {
```

Avoid:

* async inside Promise constructor

Can create double promise confusion.

---

# 1️⃣2️⃣ Executor Runs Immediately

Huge gotcha.

---

# Example

```js id="r44"
new Promise(() => {
  console.log("NOW");
});
```

prints immediately.

NOT async.

---

# 1️⃣3️⃣ Promise Methods Return NEW Promises

VERY IMPORTANT.

---

# Example

```js id="r45"
const p2 = p1.then(...)
```

Each chain creates:

* new promise

---

# Why chaining works

Each step returns:

* next promise

---

# ⚡ 2-Line Revision

> `resolve` and `reject` are normal functions provided by the Promise constructor to settle a promise as fulfilled or rejected.
> `.catch(reject)` works because functions are first-class citizens in JavaScript, allowing direct function references instead of wrapper callbacks.
