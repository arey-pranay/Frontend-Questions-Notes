# 📄 Arrow Functions, `return`, Parentheses, Curly Braces & JSX Mapping — Complete React/JS Notes

This confusion is EXTREMELY common.

Because JavaScript arrow functions have:

* implicit return
* explicit return
* object-return syntax weirdness
* JSX shorthand
* callback variations

And React uses these patterns EVERYWHERE:

* `map`
* `filter`
* `setState`
* event handlers
* hooks
* reducers

---

# 🧠 CORE RULE

Arrow functions have TWO styles:

---

# 1️⃣ Implicit Return

NO curly braces.

Automatically returns expression.

```js id="s11"
const add = (a, b) => a + b;
```

Equivalent to:

```js id="s12"
const add = (a, b) => {
  return a + b;
};
```

---

# 2️⃣ Explicit Return

Uses curly braces.

You MUST write `return`.

```js id="s13"
const add = (a, b) => {
  return a + b;
};
```

---

# 🚨 Golden Memory Rule

```text id="s14"
No braces → auto return
Braces → must use return
```

---

# 🔥 Your Example 1

```jsx id="s15"
setTodos((prev) => {
  return prev.filter((t, i) => i != index);
});
```

---

# Why `return` Needed?

Because you used:

```js id="s16"
{
}
```

Curly braces.

So this becomes:

* function BODY

and function body requires:

* explicit `return`

---

# Can Be Shortened To

```jsx id="s17"
setTodos(prev =>
  prev.filter((t, i) => i !== index)
);
```

Implicit return.

---

# 🔥 Your Example 2

```jsx id="s18"
{todos.map((todo, i) => {
  return (
    <ToDo
      key={i}
      index={i}
      todo={todo}
      onDelete={onDeleteId}
    />
  );
})}
```

Correct because:

* curly braces used
* explicit return required

---

# Can Be Shortened

```jsx id="s19"
{
  todos.map((todo, i) => (
    <ToDo
      key={i}
      index={i}
      todo={todo}
      onDelete={onDeleteId}
    />
  ))
}
```

---

# 🚨 VERY IMPORTANT JSX RULE

---

# Parentheses `()`

In JSX usually mean:

```text id="s20"
grouping expression
```

NOT function body.

---

# So This

```jsx id="s21"
x => (
  <div>Hello</div>
)
```

is:

* implicit return

---

# While This

```jsx id="s22"
x => {
  return <div>Hello</div>;
}
```

is:

* explicit return

---

# 🚨 HUGE INTERVIEW POINT

These are equivalent:

---

# Implicit

```js id="s23"
x => x * 2
```

---

# Explicit

```js id="s24"
x => {
  return x * 2;
}
```

---

# 🔥 WHY JSX USES `()`

Mostly:

* readability
* multiline formatting

---

# Example

```jsx id="s25"
() => (
  <Component />
)
```

Cleaner than:

```jsx id="s26"
() => <Component />
```

---

# 1️⃣ Object Return Gotcha

VERY IMPORTANT.

---

# ❌ WRONG

```js id="s27"
const fn = () => { name: "Pranay" };
```

Returns:

```text id="s28"
undefined
```

---

# Why?

JS thinks:

```js id="s29"
{
  name: "Pranay"
}
```

is:

* function block
* NOT object literal

---

# ✅ Correct

Wrap object in parentheses:

```js id="s30"
const fn = () => ({
  name: "Pranay"
});
```

---

# 🚨 Golden Rule

```text id="s31"
Returning object implicitly?
Wrap in ()
```

---

# 2️⃣ JSX Inside `.map()`

VERY IMPORTANT React pattern.

---

# Common Style

```jsx id="s32"
arr.map(item => (
  <Component />
))
```

Implicit JSX return.

---

# Multi-line Logic?

Use braces.

---

# Example

```jsx id="s33"
arr.map(item => {

  if (!item.active) {
    return null;
  }

  return <Component />;
})
```

---

# 🚨 Important

Once logic gets complex:

* use `{}` + explicit return

---

# 3️⃣ `return null` in React

VERY important.

```jsx id="s34"
if (!show) return null;
```

Means:

* render nothing

---

# 4️⃣ Callback Function Styles

---

# Single Expression

Prefer implicit return.

```js id="s35"
arr.filter(x => x > 5)
```

---

# Complex Logic

Prefer explicit return.

```js id="s36"
arr.filter(x => {

  const valid = x > 5;

  return valid;
})
```

---

# 5️⃣ `.map()`, `.filter()`, `.reduce()`

VERY IMPORTANT differences.

---

# `.map`

Transforms array.

Must return transformed value.

```js id="s37"
arr.map(x => x * 2)
```

---

# `.filter`

Returns boolean.

```js id="s38"
arr.filter(x => x > 5)
```

---

# `.reduce`

Accumulates value.

```js id="s39"
arr.reduce((acc, x) => acc + x, 0)
```

---

# 🚨 Common Mistake

```js id="s40"
arr.map(x => {
  x * 2;
})
```

Returns:

```text id="s41"
undefined[]
```

because:

* braces used
* no return

---

# Correct

```js id="s42"
arr.map(x => {
  return x * 2;
})
```

OR:

```js id="s43"
arr.map(x => x * 2)
```

---

# 6️⃣ React State Updater Functions

VERY IMPORTANT.

---

# Example

```jsx id="s44"
setTodos(prev => [...prev, newTodo]);
```

---

# Why callback?

Because state updates may be:

* async
* batched

Using callback guarantees:

* latest state

---

# 🚨 Huge Interview Point

Prefer updater callback when new state depends on previous state.

---

# BAD

```jsx id="s45"
setCount(count + 1)
```

Can become stale.

---

# BETTER

```jsx id="s46"
setCount(prev => prev + 1)
```

---

# 7️⃣ JSX Curly Braces `{}`

Inside JSX:

```jsx id="s47"
<div>{value}</div>
```

means:

```text id="s48"
evaluate JavaScript expression
```

---

# NOT object literal necessarily.

---

# Example

```jsx id="s49"
<div>{2 + 2}</div>
```

renders:

* 4

---

# 8️⃣ JSX Returning Arrays

Allowed.

```jsx id="s50"
return [<A />, <B />];
```

---

# 9️⃣ Parentheses in JS

Can mean MANY things:

---

# Function Call

```js id="s51"
fn()
```

---

# Grouping

```js id="s52"
(1 + 2) * 3
```

---

# Implicit Return Wrapper

```js id="s53"
x => (
  <div />
)
```

---

# Object Literal Disambiguation

```js id="s54"

  x: 1

```

# 🔟 React Rendering Rules

React can render:

* strings
* numbers
* JSX
* arrays
* null

---

# But NOT plain objects

```jsx id="s55"
<div>{{}}</div>
```

❌ Error

---

# 1️⃣1️⃣ Ternary Rendering

VERY common.

```jsx id="s56"
{
  loading
    ? <Spinner />
    : <Content />
}
```

---

# 1️⃣2️⃣ `&&` Rendering

```jsx id="s57"
{
  isAdmin && <AdminPanel />
}
```

---

# 🚨 Important Gotcha

```jsx id="s58"
0 && <X />
```

renders:

* 0

because React renders numbers.

---

# Safer

```jsx id="s59"
Boolean(count) && <X />
```

---

# 1️⃣3️⃣ Returning Multiple JSX Elements

Need:

* wrapper
  OR
* fragment

---

# Fragment

```jsx id="s60"
<>
  <A />
  <B />
</>
```

---

# 1️⃣4️⃣ Why Parentheses Used Around JSX Returns

Not mandatory technically.

But improves:

* readability
* automatic semicolon safety

---

# Example

```js id="s61"
return
<div />
```

becomes:

```js id="s62"
return;
```

due to automatic semicolon insertion.

---

# Safer

```js id="s63"
return (
  <div />
)
```

# ⚡ 2-Line Revision

> Arrow functions use implicit return without `{}` and require explicit `return` when curly braces create a function body.
> React heavily uses these patterns in `.map`, state updaters, JSX rendering, and callback functions, making return behavior essential for frontend development.
