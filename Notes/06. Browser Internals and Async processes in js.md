# 1️⃣ JavaScript Runtime Architecture

JavaScript itself is:

* single-threaded
* synchronous by default

But browsers provide:

* Web APIs
* event loop
* queues
* rendering engine

---

# 🔥 High-Level Runtime

```text id="h11"
Browser
 ├── Call Stack
 ├── Web APIs
 ├── Callback Queue
 ├── Microtask Queue
 ├── Event Loop
 └── Rendering Engine
```

---

# 2️⃣ Call Stack

Tracks currently executing functions.

---

# Example

```js id="h12"
function a() {
  b();
}

function b() {
  console.log("Hello");
}

a();
```

Stack:

```text id="h13"
a()
b()
console.log()
```

LIFO structure.

---

# 🚨 Stack Overflow

Infinite recursion:

```js id="h14"
function recurse() {
  recurse();
}
```

Causes:

```text id="h15"
Maximum call stack exceeded
```

---

# 4️⃣ Event Loop

Coordinates async execution.

---

# Simplified Flow

```text id="h17"
Call Stack empty?
    ↓ yes
Microtasks?
    ↓
Macrotasks?
```

---

# 🚨 VERY IMPORTANT RULE

## Microtasks ALWAYS run before macrotasks.

---

# 5️⃣ Microtask Queue

High-priority async queue.

---

# Examples

```js id="h18"
Promise.then()
queueMicrotask()
MutationObserver
```

---

# Example

```js id="h19"
console.log(1);

Promise.resolve().then(() => {
  console.log(2);
});

console.log(3);
```

Output:

```text id="h20"
1
3
2
```

Because promise callback enters microtask queue.

---

# 6️⃣ Macrotask Queue (Callback Queue)

Lower priority async tasks.

---

# Examples

```js id="h21"
setTimeout
setInterval
MessageChannel
I/O
```

---

# Example

```js id="h22"
setTimeout(() => {
  console.log("timeout");
}, 0);
```

Still waits until:

* current stack empty
* microtasks completed

---

# 🔥 Famous Interview Example

```js id="h23"
console.log(1);

setTimeout(() => console.log(2));

Promise.resolve().then(() => console.log(3));

console.log(4);
```

Output:

```text id="h24"
1
4
3
2
```

---

# 🚨 Why?

Order:

1. sync code
2. microtasks
3. macrotasks

# 🔥 Reflow vs Repaint

---

# Reflow (Layout)

Expensive.

Triggered when layout changes.

Examples:

* width
* height
* font-size

---

# Repaint

Visual-only changes.

Examples:

* background color
* visibility

Cheaper.

---

# 🚨 Performance Tip

Avoid layout thrashing:

```js id="h26"
element.style.width = ...
element.offsetWidth
element.style.width = ...
```

Forces repeated reflows.

# 8️⃣ Browser Threads

Browser itself is multithreaded.

---

# Common Threads

| Thread         | Purpose          |
| -------------- | ---------------- |
| JS Thread      | execute JS       |
| UI Thread      | rendering        |
| Network Thread | requests         |
| GPU Thread     | compositing      |
| Worker Threads | background tasks |

---

# 🚨 Important

JS code runs on:

* ONE main thread

This is why blocking loops freeze UI.

---

# Example

```js id="h27"
while(true) {}
```

Blocks rendering.

---

# 9️⃣ Concurrency vs Parallelism

VERY IMPORTANT interview topic.

---

# Concurrency

Tasks make progress together.

Not necessarily simultaneously.

---

# Parallelism

Tasks literally run simultaneously on multiple CPUs/cores.

---

# Example

---

# Concurrency

```text id="h28"
Task A
Task B
Task A
Task B
```

Switching between tasks.

---

# Parallelism

```text id="h29"
CPU 1 -> Task A
CPU 2 -> Task B
```

True simultaneous execution.

---

# 🚨 JavaScript is Concurrent, Not Parallel (mostly)

Main thread:

* single-threaded concurrency

Parallelism possible via:

* Web Workers
* browser internals

---

# 🔟 Web Workers

Background JS threads.

---

# Use Cases

* image processing
* heavy calculations
* ML
* parsing

---

# Example

```js id="h30"
const worker = new Worker("worker.js");
```

---

# 🚨 Important

Workers:

* cannot access DOM
* communicate via messages


# ⚡ 2-Line Revision

> JavaScript is single-threaded and asynchronous behavior is enabled through browser Web APIs, the event loop, and task queues.
> Microtasks (Promises) execute before macrotasks (`setTimeout`), while browsers achieve concurrency using multiple internal threads and APIs.
