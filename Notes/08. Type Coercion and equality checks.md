# 📄 JavaScript Type Coercion & Equality Weirdness — Frontend Interview Master Notes

This is one of the most infamous parts of JavaScript.

Interviewers LOVE these because they test:

* coercion rules
* truthy/falsy understanding
* equality internals
* primitive conversion
* object conversion
* runtime behavior

Most JS weirdness comes from:

* implicit type coercion
* objects converting to primitives

---

# 🔥 Your Example

---

# Case 1

```js id="k11"
if ([]) console.log(1);
else console.log(2);
```

Output:

```text id="k12"
1
```

---

# Why?

Because:

```js id="k13"
[]
```

is an object.

And ALL objects are:

```text id="k14"
truthy
```

even:

* `[]`
* `{}`

---

# 🚨 IMPORTANT RULE

Only these are falsy:

```js id="k15"
false
0
-0
0n
""
null
undefined
NaN
```

EVERYTHING else is truthy.

# 🚨 GOLDEN RULE

```text id="k29"
Objects are truthy,
but equality causes coercion.
```

These are DIFFERENT systems.

---

# 1️⃣ Truthy/Falsy System

Used in:

* `if`
* `while`
* `&&`
* `||`

ONLY checks truthiness.

---

# 2️⃣ Equality Coercion System

Used in:

* `==`

Performs conversions.

Different internal algorithms.

# 🔥 Loose Equality Internal Rules

---

# If types same

Compare directly.

---

# If object involved

Convert object to primitive.

---

# If boolean involved

Convert boolean to number.

---

# If string-number

Convert string to number.

# ⚡ 2-Line Revision

> JavaScript `if` conditions use truthiness (`ToBoolean`) while `==` uses complex coercion rules (`Abstract Equality Comparison`).
> Objects like arrays are always truthy, but loose equality can coerce them into primitives, causing famously weird results.
